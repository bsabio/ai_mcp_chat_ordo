# Sprint 3 Release Verification

## Purpose

This guide defines the Sprint 3 release-facing QA flow for staging canaries and durable evidence artifacts. It is intended to answer four questions for each release candidate:

1. which environment was tested
2. which canary scenarios ran
3. which evidence artifacts were produced
4. whether the release is approved, conditional, or blocked

## Preconditions

Before running the flow, make sure these inputs exist:

1. `.env.local` or equivalent shell environment contains a valid Anthropic key
2. `EVAL_LIVE_ENABLED=true`
3. `EVAL_TARGET_ENV=staging`
4. `EVAL_DEPLOYED_BASE_URL` points at the intended deployed target; `EVAL_STAGING_BASE_URL` remains supported as a legacy fallback
5. the repo has already produced `release/manifest.json` through the standard release preparation step

## Command Sequence

Run the full release-facing flow in this order.

```bash
npm run release:prepare
npm run admin:health
EVAL_LIVE_ENABLED=true EVAL_TARGET_ENV=staging EVAL_DEPLOYED_BASE_URL=https://www.studioordo.com node --env-file=.env.local --import tsx scripts/run-staging-canary.ts
node --env-file=.env.local --import tsx scripts/generate-release-evidence.ts
```

Equivalent one-shot command:

```bash
EVAL_LIVE_ENABLED=true EVAL_TARGET_ENV=staging EVAL_DEPLOYED_BASE_URL=https://www.studioordo.com npm run qa:sprint-3
```

Optional explicit env-file override:

```bash
EVAL_LIVE_ENABLED=true EVAL_TARGET_ENV=staging EVAL_DEPLOYED_BASE_URL=https://www.studioordo.com npm run qa:sprint-3 -- --env-file .env.local
```

Optional scoped canary run:

```bash
EVAL_LIVE_ENABLED=true EVAL_TARGET_ENV=staging EVAL_DEPLOYED_BASE_URL=https://www.studioordo.com node --env-file=.env.local --import tsx scripts/run-staging-canary.ts --scenario organization-buyer-funnel --scenario mcp-tool-choice-and-recovery
```

Optional manual review annotations:

```bash
node --env-file=.env.local --import tsx scripts/generate-release-evidence.ts \
  --manual-check "Founder dashboard review completed." \
  --warning "One non-blocking copy issue remains tracked outside this release."
```

Role coverage spot check:

```bash
npm run qa:roles
```

This verifies the core user-role surfaces remain covered for `ADMIN`, `AUTHENTICATED`, and `ANONYMOUS` behavior, including prompt policy, tool RBAC, dashboard visibility, and workflow continuity.

## Produced Artifacts

The Sprint 3 flow writes these machine-readable artifacts under `release/`:

1. `manifest.json`: release metadata generated by the existing release step
2. `canary-summary.json`: per-scenario deployed-canary outcomes, scorecards, failure reasons, selected model, and target URL
3. `qa-evidence.json`: manifest, health, and canary evidence plus release review status and reviewer notes

## Review Outcomes

Use the generated `qa-evidence.json` status as the default review outcome.

1. `approved`: release manifest is present, health is green, staging canaries all pass, and no warnings or manual checks remain open
2. `conditional`: required evidence is present and passing, but warnings or manual review items still need explicit acknowledgement
3. `blocked`: release manifest is missing, health fails, canary evidence is missing, or one or more canary scenarios fail

## Reviewer Checklist

Mark the candidate complete only when all required checks are satisfied.

- [ ] `release/manifest.json` exists and matches the intended release candidate
- [ ] `release/canary-summary.json` exists and records the correct deployed target URL and scenario set
- [ ] `release/qa-evidence.json` exists and the status is appropriate for the actual run outcome
- [ ] all required buyer, learner, development, and MCP canary scenarios were run or explicitly waived
- [ ] any warning or manual-check entry in `qa-evidence.json` is acknowledged by the reviewer
- [ ] any partial failure has a concrete remediation or waiver note before approval

## Failure Handling

Treat these conditions as fail-closed.

1. missing canary or QA artifact
2. health status `error`
3. canary result contains failed scenarios
4. reviewer relies on terminal output instead of durable artifacts
5. a required manual-only validation was performed but never recorded in the evidence file
